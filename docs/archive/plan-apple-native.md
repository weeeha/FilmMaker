## 0. Cross-Cutting Concerns

These decisions span all phases and must be established in Phase 0. Any change to these is a foundation-level change that should cascade through subsequent phase work.

### 0.1 Project & target layout

Xcode workspace with a single App target plus SwiftPM-backed local modules:

```
FilmPlanningApp.xcworkspace/
├── FilmPlanningApp/                    ← App target (multiplatform: macOS/iPadOS/iOS)
│   └── FilmPlanningAppApp.swift
└── Packages/
    ├── FPCore/                         ← Pure Swift domain model, no UI, no platform deps
    ├── FPPersistence/                  ← SwiftData + .filmproj document package
    ├── FPRendering/                    ← RenderEngine protocol + RealityKit/SceneKit impls
    ├── FPTimeline/                     ← Timeline model, playhead, scrubbing, animation curves
    ├── FPExport/                       ← AVAssetWriter, PDFKit, CSV export
    ├── FPAR/                           ← ARKit virtual camera + LiDAR scan
    ├── FPSync/                         ← CloudKit sync + sharing
    ├── FPAssetLibrary/                 ← Built-in catalog, user uploads, AI prop provider
    ├── FPScreenplay/                   ← .fdx import parser
    └── FPUI/                           ← Shared SwiftUI views, design system
```

Rationale: modules keep compile times sane, enforce dependency direction (`FPCore` depends on nothing; UI depends on everything), and let us write per-module tests without booting the app.

Dependency direction (strictly enforced):
```
FPUI → {FPRendering, FPTimeline, FPExport, FPAR, FPSync, FPAssetLibrary, FPScreenplay} → FPPersistence → FPCore
```

### 0.3 Data model & migrations

SwiftData schema is versioned from commit #1. Strategy:

- Every `@Model` lives in `FPPersistence/Sources/FPPersistence/Schema/Vx/`.
- A `SchemaMigrationPlan` type (`FilmProjSchemaMigrationPlan`) is created in Phase 0 with a single `V1` stage — future phases add `V2`, `V3`, etc., never edit `V1` models in place.
- `ModelContainer` is always constructed with the explicit migration plan; lightweight migrations are preferred, custom stages used only when data transforms are required.
- Every schema bump ships with a migration unit test using a fixture `.filmproj` generated by the previous version's tests.

### 0.4 Render engine abstraction (`RenderEngine` protocol)

Renderer is hidden behind a protocol so RealityKit is the default and SceneKit remains a viable fallback (and, per design Section 10 open question #1, so Unity/Unreal embed stays a future option).

```swift
// FPRendering/Sources/FPRendering/RenderEngine.swift
public protocol RenderEngine: AnyObject {
    associatedtype ViewRepresentable  // SwiftUI view handle for the live viewport
    func makeViewport(scene: SceneGraph) -> ViewRepresentable
    func load(entity: StageEntity) async throws
    func apply(transform: Transform, to entityID: UUID)
    func snapshot(at time: CMTime, size: CGSize) async throws -> CVPixelBuffer
    func tearDown()
}
```

Two implementations shipped over time:
- `RealityKitEngine` — primary, lives in `FPRendering/Sources/FPRendering/RealityKit/`
- `SceneKitEngine` — fallback, lives in `FPRendering/Sources/FPRendering/SceneKit/`, stub-complete by end of Phase 1, feature-complete only if RealityKit blocks us on a platform

App wiring picks the engine via a single factory (`RenderEngineFactory`) based on OS version and a developer override defaults key.

### 0.5 Document package layout (`.filmproj`)

File format is fixed in Phase 0 and treated as a compatibility contract going forward (see design Section 3). `FPPersistence` owns a `FilmProjectDocument: FileDocument` that reads/writes the package atomically, with per-subfolder accessors so the rendering cache (`Renders/`) can be invalidated without touching the SwiftData store.

UTI registration (Info.plist): `com.nickv.filmplanning.filmproj`, conforming to `com.apple.package`. Declared in Phase 0 so file associations work from day one.

### 0.6 App state layer

Single source of truth using `@Observable` (Swift 5.9+):

- `ProjectStore` — opened project, active selection, undo/redo manager
- `TimelineStore` — playhead, playback state, scrubbing
- `SelectionStore` — currently selected entities across scene and timeline
- `RenderSettingsStore` — viewport quality, preview resolution

Stores are injected through `@Environment` — no singletons, no global state. Every store is independently testable with fixture data.

### 0.7 Error & logging conventions

- Structured logging via `os.Logger`, one subsystem per module (`com.nickv.filmplanning.rendering`, `.persistence`, `.ar`, `.sync`, `.export`).
- Errors are typed per module (`RenderingError`, `PersistenceError`, `ARCaptureError`, etc.), all conforming to a shared `FPError` protocol with `userDescription`, `debugDescription`, and `recoverySuggestion`.
- Top-level error presentation is centralized in `FPUI/ErrorPresenter` — views throw into it, never surface `Error` directly.
- No `fatalError` in shipping code paths except truly-impossible invariant violations; prefer logged + recovered failure.

### 0.8 Testing strategy

Per the design's non-functional constraints, we need regressions caught early across three surfaces:

| Layer | Framework | What we test |
|---|---|---|
| Unit | Swift Testing (`import Testing`) | `FPCore` domain logic, `FPPersistence` migrations, `FPScreenplay` parser, timeline math |
| Integration | Swift Testing + XCTest | SwiftData round-trips, CloudKit mock sync, `.filmproj` read/write, export pipeline on a fixture project |
| UI | XCUITest (smoke) + SwiftUI previews | Critical flows: open project, add entity, scrub timeline, export storyboard |
| Snapshot | `swift-snapshot-testing` | Storyboard frames, PDF exports, timeline ruler rendering |
| Performance | XCTest measure blocks | 60 fps viewport budget, export throughput, SwiftData fetch on 500-shot fixture |

Each phase's exit criteria includes test coverage for the new surface area. Fixture projects (`Fixtures/small.filmproj`, `Fixtures/large.filmproj`) live in a test-support package.

### 0.9 Tooling & CI

- Swift package graph builds on every commit; one scheme per platform (`FilmPlanningApp-macOS`, `-iPadOS`, `-iOS`).
- SwiftLint + swift-format (Apple's) with a shared config in repo root.
- GitHub Actions (or local Xcode Cloud) runs build + unit + snapshot tests on PR. `[NEEDS CLARIFICATION: CI provider preference — Xcode Cloud, GitHub Actions, or self-hosted?]`

## Phase 5 — AR Virtual Camera + LiDAR Scanning

**Goals**
- Virtual-camera mode on iPhone/iPad: device pose streams to a scene camera, records as a take.
- LiDAR scan mode: `ARMeshAnchor` capture → USDZ → dropped into scene as `.scan` entity.
- Take playback: recorded pose curves scrub with timeline.

**Deliverables**
- `FPAR.VirtualCameraSession` wrapping `ARSession` with `ARWorldTrackingConfiguration`.
- `FPAR.LiDARScanSession` with `.meshWithClassification` scene reconstruction.
- Pose-sampling pipeline at 60 Hz → `AnimationCurve` → stored on shot.
- Mac is read-only for AR features; receives scans/takes from iOS via CloudKit (Phase 6) or airdropped `.filmproj`.

**Key files/modules to create**
- `Packages/FPAR/Sources/FPAR/VirtualCameraSession.swift`
- `Packages/FPAR/Sources/FPAR/LiDARScanSession.swift`
- `Packages/FPAR/Sources/FPAR/MeshBaker.swift` — `ARMeshAnchor` → `MDLAsset` → USDZ
- `Packages/FPAR/Sources/FPAR/PoseSampler.swift`
- `Packages/FPRendering/Sources/FPRendering/Entities/ScanEntity.swift`
- `Packages/FPRendering/Sources/FPRendering/MeshDecimation.swift` — low-poly proxy generator
- `Packages/FPUI/Sources/FPUI/AR/VirtualCameraView.swift`
- `Packages/FPUI/Sources/FPUI/AR/ScanCaptureView.swift`

**SDK touchpoints**
- ARKit: `ARSession`, `ARWorldTrackingConfiguration`, `ARMeshAnchor`, `ARCamera`
- Model I/O for mesh baking
- `AVCaptureSession` if we want video pass-through recording alongside pose

**Risks**
- LiDAR only on Pro-tier iPhones/iPads (design Section 5). UI must clearly gate the feature.
- Large meshes + edit-time perf: proxy decimation must be default, full mesh only at export.
- Thermal throttling on long AR sessions — surface warnings, checkpoint pose data frequently.

**Exit criteria**
- Recording a 30-second virtual-camera take and replaying it through the timeline is frame-accurate.
- A room-sized LiDAR scan imports and renders at the agreed fps target with the low-poly proxy active.
- AR features are fully gated and labeled as iOS-only in the UI on macOS.

## Phase 6 — CloudKit Sync + Sharing

**Goals**
- SwiftData entities mirror to CloudKit private DB automatically.
- Large blobs (USDZ, scans, audio) sync as `CKAsset` records, lazy-downloaded.
- Share a project via `CKShare` with read-only access by default; edit role opt-in.
- Sync is best-effort and never blocks local editing.

**Deliverables**
- `FPSync.CloudKitContainerManager` setting up the mirrored `ModelContainer`.
- `FPSync.AssetSyncCoordinator` for `CKAsset` upload/download of package blobs.
- `FPSync.ShareCoordinator` for `CKShare` creation and participant invite flows.
- CRDT-lite timeline ordering (stable insertion-ordered IDs) made authoritative for conflict resolution (the IDs were reserved in Phase 2 — now they're used).
- Share-accept flow on all three platforms.

**Key files/modules to create**
- `Packages/FPSync/Sources/FPSync/CloudKitContainerManager.swift`
- `Packages/FPSync/Sources/FPSync/AssetSyncCoordinator.swift`
- `Packages/FPSync/Sources/FPSync/ShareCoordinator.swift`
- `Packages/FPSync/Sources/FPSync/ConflictResolver.swift`
- `Packages/FPPersistence/Sources/FPPersistence/Schema/V{n}/` — add CloudKit-compatible constraints (no unique constraints, optional relationships, etc.) as a new schema version if V1 didn't anticipate them
- `Packages/FPUI/Sources/FPUI/Sharing/ShareProjectSheet.swift`

**SDK touchpoints**
- CloudKit: `CKContainer`, `CKDatabase`, `CKShare`, `CKAsset`, `CKSyncEngine` (or SwiftData's built-in mirror)
- SwiftData `ModelConfiguration(cloudKitDatabase:)`

**Risks**
- CloudKit requires every SwiftData model to have optional relationships and no unique constraints — if V1 didn't follow those rules we'll ship a migration here. Audit V1 during Phase 0 to avoid surprise work.
- `CKAsset` size/throughput limits for 5 GB projects; may need a chunked upload queue.
- Initial sync of a large project is slow — need a progress UI and conflict-free retry logic.

**Exit criteria**
- Opening the same project on Mac and iPad reflects edits within 60 seconds end-to-end on a typical connection.
- Read-only share link renders the project on a second iCloud account with no edit affordances.
- Losing network mid-edit never corrupts the local store; reconnection drains the queue.
